http有哪些方法

需要了解每个方法的具体作用
1、http1.0规定了三种请求防范：get、post、head
2、http1.1新增加五种请求方法：put、delete、options、trace、connect

get和post有什么区别

1、数据传输方式不同：Get请求通过URL传输数据，post通过请求体传输
2、安全性不同：post的数据由于咋请求主题内，所有有一定的安全性保证；而get的数据再URL中，容易泄露
3、数据类型不同：GET只允许ASCII字符；而post无限制
4、GET无害：刷新、后退等浏览器操作get请求时无害的；而post可能重复提交表单
5、特性不同：GET是安全且幂等，而post是非安全非幂等的

put和post都是给服务器发送新增资源，有什么区别

都能创建资源，根本区别：
    1、PUT方法是幂等的，一般PUT的URI指向是具体单一资源
    2、POST方法是非幂等，POST可以指向资源集合
ps：『POST表示创建资源，PUT表示更新资源』这种说法是错误的，两个都能创建资源，根本区别就在于幂等性

PUT和PATCH都是给服务器发送修改资源，有什么区别

都能更新资源，根本区别
    put是对单一资源整体更新
    patch已知资源进行局部更新

200和304的理解和区别、202

200是指浏览器没有跟服务器确认，直接用了浏览器缓存
304是浏览器和服务器多确认了一次缓存有效性，再用的缓存
304 Not Modified 比 200 OK (from cache) 慢，指的是浏览器还向服务器确认了下 “If-Not-Modified”，才用的缓存

同样是重定向307，303，302的区别

302是http1.0的协议状态码，表示资源重定向
在http1.1版本为了细化302状态码新增了303和307

  • 303：会将post请求变坏get请求进行重定向
  • 307：取代原来302状态码，尊重浏览器标准，依旧使用post进行重定向

http协议栈是怎么样的？

HTTP是应用层协议。它把联网的细节都交给了通用、可靠的因特网传输协议TCP\IP协议。
HTTP(应用层)-TCP(传输层)-IP(网络层)-数据链路层-物理层

HTTP的keep-alive的作用

什么是keep-alive
Keep-Alive模式又称持久连接、连接重用，通过在http请求头中加入Keep-Alive功能使客户端到服务器端的连接持续有效，当出现对服务器的后继请求时，Keep-Alive功能避免了建立或者重新建立连接。
http 1.0中默认是关闭的，需要在http头加入”Connection: Keep-Alive”，才能启用Keep-Alive；http 1.1中默认启用Keep-Alive，如果加入”Connection: close “，才关闭。目前大部分浏览器都是用http1.1协议，也就是说默认都会发起Keep-Alive的连接请求了，所以是否能完成一个完整的Keep-Alive连接就看服务器设置情况
作用
TCP连接减少了，降低拥塞控制，减少主机和路由器上的CPU和内存开销
无需再进行握手，减少了后续请求的延迟
报告错误无需关闭TCP连

Keep-Alive模式时，客户端如何判断请求所得到的响应数据已经接收完成

  • 使用消息首部字段Conent-Length
  • 分块编码（Transfer-Encoding: chunked。最后一块长度值必须为 0，对应的分块数据没有内容，表示实体结束

TCP的三次握手和四次挥手

三次握手：
1、第一次握手：客户端给服务器发送一个 SYN (tcp标志，synchronize，使…..同步)报文。
2、第二次握手：服务器收到 SYN 报文之后，会应答一个 SYN+ACK 报文。
3、第三次握手：客户端收到 SYN+ACK 报文之后，会回应一个 ACK 报文。
4、服务器收到 ACK 报文之后，三次握手建立完成。
四次挥手：
1、第一次挥手：客户端发送一个 FIN（finish） 报文，报文中会指定一个序列号。
2、第二次握手：服务端收到 FIN 之后，会发送 ACK 报文，且把客户端的序列号值 + 1 作为 ACK 报文的序列号值，表明已经收到客户端的报文了。
3、第三次挥手：如果服务端也想断开连接了，和客户端的第一次挥手一样，发给 FIN 报文，且指定一个序列号。
4、第四次挥手：客户端收到 FIN 之后，一样发送一个 ACK 报文作为应答，且把服务端的序列号值 + 1 作为自己 ACK 报文的序列号值。服务端收到自己的 ACK 报文。

浏览器如何根据http的url地址，显示远端服务器的某个简单html资源【重点】

关键：域名解析服务DNS
1、浏览器从url中解析出服务器的主机名
2、浏览器向本地DNS服务器请求解析该域名，解析为真实IP地址
3、浏览器与目标服务器建立一条tcp连接，发起TCP握手(3次)
4、握手成功，构造request，浏览器向服务器发送http请求报文
5、服务器向浏览器回送一条http响应报文
6、关闭连接。浏览器显示报文

http和https的区别

http是明文传输协议，交互过程以及数据传输都没有进行加密，通信双方也没有进行任何认证，所以对于一些敏感信息的传输就很不安全。
HTTPS：是HTTP的安全版，具有安全性的基于ssl加密的传输协议。

https如何保障安全的

关键：理解对称加密、非对称加密、CA证书、数字签名
https使用SSL安全协议来保障安全性。具体体现在密钥和CA证书验证上。
1、首先服务端需要将自己的公钥发给证书颁发机构，用于申请CA证书
进入正式通信流程：
2、浏览器发出安全请求，服务器返回CA证书
3、浏览器验证证书
使用同样的hash算法将证书的除数字签名外的信息生成消息摘要，
同时使用CA的公钥解密证书上的数字签名，返回消息摘要
判断是否相同，相同则浏览器使用机构公钥解析，获取服务端的公钥，是则断开连接
4、浏览器使用服务器的公钥加密自己的随机生成的对称密钥、发送给服务器
5、服务器使用私钥解密，获取对称秘钥
6、后续双方便使用对称秘钥来加密通信
PS:
证书颁发机构自己也有一对公钥私钥。机构利用自己的私钥来加密服务器的公钥，并且通过服务端网址等信息生成一个证书签名，证书签名同样经过机构的私钥加密。证书制作完成后，机构把证书发送给了服务端
CA证书中包括：签发者、服务端网址、服务器公钥、服务器私钥、证书到期时间以及数字签名等
数字签名：即签发者、服务端网址、服务器公钥、服务器私钥、证书到期时间根据hash算法返回一个消息摘要，再经过CA私钥加密返回数字签名

cookie、session、token(不要与LocalStorage和SessionStorage混淆)（如何保存登录状态的）

关键的理解：
session实现和session是不同的，可将session实现理解为狭义的session，session理解为广义的session。

  • 广义的session
    首先session是一个抽象的概念。为了实现中断和继续等操作，将浏览器与server之间一对一的交互，抽象为“会话”，进而衍生出“会话状态”，对于如何维持这种会话，没有要求
  • 狭义的session
    基于cookie来实现用户和数据的映射。为了绕开cookie的各种限制，借助cookie本身和服务端存储实现的，一种更高级的会话状态的实现。服务器存储了一些必须的用户信息，并在用户请求时返回存储的信息例如session Id，浏览器存储在cookie[仓库]，在后续的会话通过session Id保持状态
    流程
    1、客户端**带着用户信息**发送请求，服务器端收到后校验用户信息，校验正确就会在服务器端存储一个sessionId和session的映射关系[用户的所有关键信息]
2、服务器端返回response，并且将sessionId以set-cookie的方式种在客户端，sessionId就存在了客户端。这里要注意的是，将sessionId存在cookie并不是一种强制的方案，而是大家一般都这么做，而且发请求的时候符合domain和path的时候，会自动带上cookie，省去了手动塞的过程。
3、客户端继续发起请求，服务端通过cookie中的sessionId找到对应的session来识别用户
4、返回session
  • 承上启下的解释： session 有如用户信息档案表, 里面包含了用户的认证信息和登录状态等信息. 而 cookie 就是用户通行证。很少单独使用cookie，而是使用session(已结合cookie的优势)

  • cookie
    cookie是session的一种具体实现方案，通常是服务器在http响应中设置set-cookie字段来设置。当然也能通过js在前端生成。浏览器会把cookie信息片段以键值对存储在客户端的cookie”仓库“。
    后续浏览器的需求会带上存储在本地的cookie，继而保持会话状态
    流程

    1. 浏览器先发送请求，服务器的应答包头中返回set-cookie
2、从 Web 服务器的应答包头中提取所有的 cookie。
2. 解析这些 cookie 的组成部分（名称，值，路径等等）。
3. 判定主机是否允许设置这些 cookie。允许的话，则把这些 Cookie 存储在本地
4. 后续，浏览器根据请求的 URL 和本地存储 cookie 的属性，判断那些 Cookie 能被发送给 Web 服务器。
5. 对于多个 cookie，判定发送的顺序。
6. 把需要发送的 Cookie 加入到请求 HTTP 包头中一起发送
7、服务器校验成功则返回请求数据，校验失败则返回错误码

    如何在跨域请求中使用cookie

    根据浏览器同源策略，不允许浏览器访问跨域的Cookie，需要服务器和浏览器同时支持

    • 服务器
      服务器端使用CROS协议解决跨域访问数据问题时，

    • 浏览器

      禁用cookie的解决方案
      1、URL重写，就是把session id直接附加在URL路径的后面
      2、在表单隐藏字段
      https://segmentfault.com/a/1190000004322487#articleHeader13

  • token
    token 也称作令牌，由uid（用户唯一身份标识）+time+sign（签名）[+固定参数]。token 的认证方式类似于临时的证书签名, 并且是一种服务端无状态的认证方式, 非常适合于 REST API 的场景. 所谓无状态就是服务端并不会保存身份认证相关的数据。每次请求时都带上token；服务器收到请求后解析token并验证相关信息，服务器校验成功则返回请求数据，校验失败则返回错误码。与前面两张方式最大的区别即是通过token的计算时间换区了session的存储空间。
    流程
    1、浏览器发送登录信息成功后，服务器返回token给客户端
2、客户端收到数据保存在本地仓库
3、客户端再次访问服务器，将token放入headers中
4、服务器端采用filter过滤器校验。校验成功则返回请求数据，校验失败则返回错误码

cookie、localStorage和SessionStorage的区别

相同：都可以用于浏览器端存储数据，均是字符串类型的键值对
区别：localStorage和SessionStorage属于webstorage，创建的目的是存储客户端数据；2️而cookie创建是为了保持http的状态
